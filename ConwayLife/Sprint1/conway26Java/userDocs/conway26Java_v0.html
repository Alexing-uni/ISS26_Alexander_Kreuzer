ConwayLife Sprint1

Introduction

Realizzazione in Java del GAME OF LIFE DI CONWAY

Requirements

    Realizzare una versione in Java del gioco Life di Conway, come gioco zero-player. 
    Il gioco consiste nell’introdurre una Griglia di Celle il cui stato (cella ‘viva’ o cella ‘morta’) 
    evolve come stabilito dallle regole di ConwayLife

    L’utente umano deve poter:
     - specificare la configurazione iniziale della griglia del gioco	
     - vedere l’evoluzione del gioco in forma opportuna 
        (si veda Problema della vista del gioco )
     - fermare e far ripartire l’evoluzione del gioco
     - pulire (a gioco fermo) la configurazione della griglia del gioco

Requirement analysis

Dal testo dei requisiti si evincono le seguenti entità e concetti fondamentali del dominio applicativo:

  • Zero-player game: Il gioco, una volta avviato, evolve autonomamente. L'utente interviene solo per configurare lo stato iniziale, avviare, mettere in pausa o pulire il sistema, ma non partecipa attivamente durante le transizioni di stato.
  • Cella (Cell): Un'entità base che possiede un singolo stato logico: può essere solo viva (true) o morta (false).
  • Griglia (Grid): Un contenitore spaziale bidimensionale e finito composto da Celle.
  • Regole di Conway: La logica matematica di business che determina lo stato futuro di ogni singola Cella basandosi sul numero di vicini vivi nella generazione corrente.

Problem analysis

Per la realizzazione del sistema, impostiamo un'architettura logica basata sui Principi SOLID, puntando a una netta separazione delle responsabilità tra il nucleo logico del gioco (core business) e l'interazione con l'utente (dispositivi di I/O).

Come si deduce dal diagramma UML architetturale:

  • Dignità Ontologica (POJOs): Introduciamo classi esplicite per Grid e Cell, evitando di "ridurre" il dominio a semplici matrici di interi o booleani. Questo rende il codice coerente con i requisiti.
  • Separazione della Business Logic: La classe Life ha l'unica responsabilità di applicare le regole del gioco sulla Grid. Non gestisce né grafica né interazioni esterne.
  • Inversione delle Dipendenze (Plug & Play): Il controllo del flusso è affidato al LifeController, il quale interagisce con l'esterno esclusivamente tramite interfacce (IOutDev e GameController). Questo rende il motore del gioco totalmente indipendente dai dettagli tecnologici; i dispositivi fisici (Console, Swing, Web) dovranno adattarsi al dominio, e non viceversa.

Test plans

Prima di scrivere il codice di produzione, definiamo un piano di testing basato sul comportamento atteso (WHAT) del sistema. Utilizzeremo JUnit per collaudare il motore logico esposto tramite una LifeInterface.

Il test principale (testOscilla) verificherà il comportamento di un pattern noto come Blinker (oscillatore di periodo 2). Configurando tre celle vive orizzontali adiacenti, ci aspettiamo che:

  1. Dopo una chiamata a nextGeneration(), le celle formino una linea verticale.
  2. Dopo una seconda chiamata a nextGeneration(), le celle tornino alla configurazione orizzontale di partenza.

Project

Il progetto è strutturato utilizzando Gradle per la gestione delle dipendenze e del processo di build. Il codice sorgente è diviso in package per rispettare l'architettura logica:

  • main.java.conway.domain: Contiene il core logico (Life, Cell, Grid, interfacce).
  • main.java.conway.devices: Contiene le implementazioni concrete di input e output (es. MockOutdev).
  • La classe principale MainConwayLifeJava agisce da entry-point per configurare le dipendenze (Dependency Injection).

Testing

La suite di test JUnit può essere eseguita tramite Eclipse o con il comando gradlew test. Inoltre, per garantire la qualità del codice, è stato integrato il plugin JaCoCo nel file build.gradle. Eseguendo gradlew test jacocoTestReport viene generato un report HTML che mostra la percentuale di Code Coverage delle classi del dominio.

Deployment

La distribuzione del sistema in questa fase (Sprint 1) avviene tramite un file archivio .jar eseguibile, che raggruppa le classi e le dipendenze in un unico pacchetto portabile.

I passi operativi sono:

  • Generazione: comando gradlew jar
  • Esecuzione autonoma: java -jar build/libs/conway26Java-1.0.jar

Maintenance

Grazie all'architettura basata su interfacce, le future implementazioni (Sprint 2: Java Swing, Sprint 3: Pagine HTML/WebSockets) richiederanno la sola aggiunta di nuovi dispositivi (package devices), lasciando completamente inalterato il package domain e la classe Life.
