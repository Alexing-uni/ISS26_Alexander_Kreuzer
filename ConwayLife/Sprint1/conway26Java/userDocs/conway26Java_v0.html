ConwayLife Sprint1

Introduction

Realizzazione in Java del GAME OF LIFE DI CONWAY

Requirements

    Realizzare una versione in Java del gioco Life di Conway, come gioco zero-player. 
    Il gioco consiste nell’introdurre una Griglia di Celle il cui stato (cella ‘viva’ o cella ‘morta’) 
    evolve come stabilito dallle regole di ConwayLife

    L’utente umano deve poter:
     - specificare la configurazione iniziale della griglia del gioco	
     - vedere l’evoluzione del gioco in forma opportuna 
        (si veda Problema della vista del gioco )
     - fermare e far ripartire l’evoluzione del gioco
     - pulire (a gioco fermo) la configurazione della griglia del gioco

Requirement analysis

Dal testo dei requisiti si evincono le seguenti astrazioni e concetti fondamentali del dominio applicativo:

  • Zero-player game: Il sistema, una volta inizializzato il suo stato interno, procede l'esecuzione in un thread autonomo. L'interazione dell'utente è limitata all'invocazione di metodi di controllo del ciclo di vita (es. start, stop, clear).
  • Cella (Cell): Astrazione del dominio modellata tramite un'interfaccia (ICell) e una classe concreta (Cell). Incapsula una singola variabile di istanza primitiva (booleana) per rappresentarne lo stato logico (viva/morta).
  • Griglia (Grid): Struttura dati aggregatrice, modellata tramite l'interfaccia IGrid. Incapsula una matrice bidimensionale di oggetti ICell, mantenendo la responsabilità della mappatura spaziale (le coordinate appartengono alla Griglia, non alla Cella).
  • Regole di Conway: La Business Logic del sistema, implementata in una classe dedicata (es. Life), che manipola le interfacce del dominio per computare la generazione successiva.

Problem analysis

Per la realizzazione del sistema, definiamo un'architettura software basata sui Principi SOLID, garantendo un forte disaccoppiamento tra il core domain e i dettagli implementativi (I/O).

Come si deduce dal diagramma UML architetturale:

  • Design orientato alle interfacce (POJOs): Definizione esplicita di tipi di dato astratti (ICell, IGrid) per garantire la "Dignità Ontologica" del dominio. Evitiamo anti-pattern come l'uso di matrici di primitivi (boolean[][]), garantendo estensibilità per futuri sviluppi.
  • Single Responsibility Principle (SRP): La classe Life incapsula esclusivamente il motore algoritmico operando su istanze di IGrid. Non espone alcun metodo relativo alla renderizzazione grafica o all'input.
  • Dependency Inversion Principle (DIP): L'orchestrazione è delegata alla classe LifeController. Le sue dipendenze (Input/Output) vengono iniettate nel costruttore sotto forma di interfacce (IOutDev, LifeInterface). I dispositivi fisici dovranno implementare questi contratti, proteggendo il dominio da cambiamenti tecnologici.

Test plans

Seguendo le logiche del testing comportamentale, utilizzeremo il framework JUnit per collaudare i metodi pubblici del dominio.

Il metodo di test principale (testOscilla) istanzia il motore logico e configura una dipendenza IGrid creando un pattern noto come Blinker (oscillatore di periodo 2). Tramite asserzioni (assertTrue, assertFalse), verificheremo che:

  1. A seguito di un'invocazione del metodo nextGeneration(), lo stato degli oggetti ICell interni muti formando una linea verticale.
  2. A seguito di una seconda invocazione, il sistema torni all'identico stato istanziato inizialmente (orizzontale).

Project

Il progetto è configurato mediante Gradle. L'alberatura dei package riflette rigorosamente la Separation of Concerns:

  • main.java.conway.domain: Contiene esclusivamente POJO e astrazioni (Life, ICell, IGrid). È un modulo totalmente agnostico rispetto all'esterno.
  • main.java.conway.devices: Contiene le classi concrete che implementano i contratti di I/O (es. MockOutdev implementa IOutDev).
  • La classe MainConwayLifeJava funge da Composition Root. È l'unico punto del sistema autorizzato a istanziare oggetti concreti (new) e a cablarli tramite Dependency Injection.

Testing

La suite di test JUnit può essere eseguita tramite IDE o con il comando CLI gradlew test. Per certificare la robustezza delle classi del dominio, è stato integrato il plugin JaCoCo nel build.gradle. Il task gradlew test jacocoTestReport genera un report HTML per la validazione della Code Coverage.

Deployment

Il build finale per questo Sprint si concretizza in un file archivio eseguibile (.jar), contenente il bytecode compilato e il file manifest.

I task Gradle associati sono:

  • Compilazione e pacchettizzazione: gradlew jar
  • Esecuzione su JVM: java -jar build/libs/conway26Java-1.0.jar

Maintenance

L'architettura basata su interfacce garantisce che i futuri aggiornamenti tecnologici (Sprint 2: Java Swing, Sprint 3: WebSocket/HTML) avverranno per mera estensione. Basterà iniettare nuove classi implementative nel package devices senza alterare in alcun modo il codice sorgente del package domain.
